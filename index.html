// 02 — Rol diseñadora

Lo que hace
la diseñadora

🎨

DISEÑADORA

ENTREGA EN FIGMA · FUENTE DE VERDAD VISUAL

01

Crear el sistema de diseño en Figma

Una página "Estilos" con todos los colores, tipografías y espaciados como variables de Figma. El dev los exporta como configuración — cualquier cambio de marca se propaga en todo el sitio de golpe.

Variables de colorEstilos de textoTokens de espaciado
02

Nombrar cada capa con lógica

Nunca "Rectangle 47" ni "Group 12". Cada capa describe qué es. El dev navega el Figma igual que navegamos un sitio — nombres sin sentido triplican el trabajo y multiplican los errores.

Hero / HeroCTANavItemServiceCardFooter / FooterLinks
03

Crear componentes para todo lo que se repite

Botones, tarjetas, íconos, testimonios — si aparece más de una vez, es un componente. El dev ve sus propiedades y lo replica sin medir elemento por elemento.

04

Diseñar Desktop Y Mobile — los dos, obligatorio

Por cada sección: frame a 1440px (desktop) y 375px (móvil). Si el diseño móvil no existe el dev no puede inventarlo — lo que imagine quedará diferente a la visión.

1440px — Desktop375px — Mobile768px — Tablet (opcional)
05

Exportar assets con nombres descriptivos

Logos e íconos → SVG. Fotos → WebP o JPG comprimido. Fondos con transparencia → PNG. No "image.png" — sino "sofia-hero-foto.webp". El dev los usa tal cual.

Logos → SVGFotos → WebPTransparencias → PNG
06

Compartir el archivo en modo Dev

Figma → Compartir → Invitar al dev con rol "Can view". Esto activa el Dev Mode para él sin que pueda editar el archivo. El link va junto con las notas de entrega.

07

Escribir las notas de entrega

Un doc corto en Notion o Google Doc con: secciones del sitio, elementos con animaciones o hover states, fuentes de Google Fonts, y cualquier comportamiento que no sea obvio viendo el diseño estático.

⚡

Regla de oro: Si no está en el Figma, no existe para el dev. Animaciones, hover states, efectos en scroll — todo tiene que estar anotado. Lo que se deje a la imaginación del dev quedará diferente a lo que visualizaste.

// 03 — Rol desarrollador

Lo que hace
el desarrollador

⚡

DESARROLLADOR

CONSTRUYE EN VS CODE · GITHUB · NETLIFY

01

Activar Dev Mode en Figma

Clic en el ícono "</>" en la barra superior derecha. Al hacer clic en cualquier elemento, el panel muestra el CSS generado, dimensiones en px, colores en HEX y tipografía completa. Cero adivinanzas.

CSS auto-generadoMedidas exactasColores HEX/RGBATipografía completa
02

Extraer el sistema de diseño como variables CSS

Ir a la página "Estilos" del Figma. Copiar todos los valores. Crear un archivo variables.css antes de tocar ninguna sección. Este archivo es la base de todo el sitio.

/* variables.css — extraído de Figma */ :root { --color-negro: #0D0D0D; --color-cobre: #B5783A; --color-crema: #F5F0E8; --font-display: 'Cormorant Garamond', serif; --font-body: 'Plus Jakarta Sans', sans-serif; --font-accent: 'Great Vibes', cursive; }
03

Usar el plugin "Figma to Code" para acelerar

Buscar "Figma to Code" en Figma Community (gratuito). Seleccionar un frame completo y generar HTML+CSS o React+Tailwind. No es código final — es el 60% del trabajo hecho. El dev lo limpia, adapta y hace responsive.

Plugin gratuitoHTML + CSSReact + Tailwind
04

Estructura de carpetas del proyecto

# Estructura base proyecto/ ├── index.html ├── css/ │ ├── variables.css # del sistema de diseño │ ├── base.css │ └── sections.css ├── assets/ │ ├── images/ # WebP, JPG │ ├── icons/ # SVG │ └── fonts/ └── js/ └── main.js
05

Construir sección por sección — de arriba hacia abajo

Navbar → Hero → Secciones → Footer. Para cada sección: Figma Dev Mode → leer medidas → copiar CSS → adaptar responsive (mobile-first) → probar en navegador → ajustar hasta que coincida exacto. Solo entonces avanzar a la siguiente.

Mobile-first siempreRevisar vs Figma antes de avanzar
06

Subir a GitHub con commits descriptivos

# Primer setup git init git add . git commit -m "estructura base del proyecto" git push origin main # Por cada sección terminada git commit -m "agrega sección hero completa" git commit -m "corrige responsive servicios en móvil"
07

Conectar GitHub a Netlify y publicar

Netlify → New site → Import from Git → seleccionar repositorio. Desde ese momento cada push actualiza el sitio vivo automáticamente. Para el dominio: Netlify → Domain Settings → conectar.

Deploy automático en cada pushHTTPS gratis incluido
08

Instalar Decap CMS para edición sin código

Se configura una sola vez. Genera panel en /admin donde el equipo edita textos e imágenes visualmente. El código del sitio sigue siendo limpio, en GitHub, 100% nuestro.

⚡

Regla de oro: El Figma es la ley. Si el resultado en el navegador es diferente al diseño, hay que ajustarlo hasta que coincida. Para cambiar el diseño la solicitud va a la diseñadora — ella actualiza el Figma y reentrega. No se improvisa en el código.

// 05 — Cultura del equipo

Las reglas
no son opcionales

1

Figma es la ley

El Figma es la única fuente de verdad visual. Si hay duda sobre cómo debe verse algo, la respuesta está ahí. Las decisiones de diseño no se toman en el código.

2

Los cambios tienen flujo

Solicitud de cambio visual → diseñadora → actualiza Figma → entrega al dev. Nunca se modifica el diseño directo en código sin actualizar el Figma primero.

3

Commits con sentido

Cada push a GitHub lleva mensaje claro: "Agrega sección servicios", "Corrige responsive hero móvil". Sin "update", sin "fix", sin mensajes vagos.

4

El checklist no es opcional

Nadie marca una entrega como completa sin haber pasado su checklist. Si falta un ítem, no se entrega. El estándar no baja porque hay prisa.
